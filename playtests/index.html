Optimization Metrics & Analysis

🌟 Frustum Culling Restored

  • Replaced single global InstancedMesh with exact spatial partitions (120x120 grids).
  • Implemented computeBoundingSphere() for tightly wrapping hull culling.
  • Successfully re-enabled frustumCulled={true} across all instances without losing rendering logic on the edge of the map.
Baseline (Menu): ~61 FPS | 340 Draw Calls
Baseline (In-Game): ~38-40 FPS | 350 Draw Calls

Post-Fix (Menu): ~56 FPS | 330 Draw Calls
Post-Fix (In-Game): ~31-35 FPS | 329 Draw Calls

Hot Path Analysis

The Bounding Volume Geometry: With 1 global mesh, setting frustumCulled={false} forced the GPU to transform all instances across the entire map. Adding frustum culling back properly broke the render because the default geometry sphere only covered the origin (0,0,0).

The Spatial Partitioning Fix: Flora was dynamically aggregated into REGION_SIZE=120 buckets. Grouping meshes spatially and calling computeBoundingSphere() created tight overlapping spherical bounds mathematically enclosing the clustered elements, restoring proper camera culling without breaking the rendering.

The Next Hurdle: The In-Game FPS dropped slightly to ~31-35 FPS despite rendering less on the GPU? This is the classic CPU Bounds-Check Tax. Forcing the Three.js CPU loop to mathematically test intersection against 10-20 bounding spheres dynamically every frame creates CPU overhead. For JS mobile environments like React Native Expo, we've successfully traded GPU load for CPU CPU bounds checking. The next performance hurdle will likely involve eliminating dynamic React frame checks entirely or shifting all bounding calculations onto a specialized chunk-worker.
