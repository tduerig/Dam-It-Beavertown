🏔️ Terrain Modification Analysis

The Cascade of Pain root cause

Every terrain modification (mud dig, mud pat, leaf decay) triggers a catastrophic cascade that blocks the main thread for 50-200ms on Android. Here's the full chain:

User Action (dig mud / pat mud / chop tree) └─► modifyTerrain(x, z, amount, radius=3) └─► { ...state.terrainOffsets } // Shallow-copy ALL keys │ ⚠ After 50 pats: 2,500+ keys copied │ ⚠ Each copy: O(N) where N = total modified cells ever │ └─► Zustand notifies ALL terrainOffsets subscribers │ ├─► Chunk.tsx × 49 chunks │ └─► useMemo([terrainOffsets]) invalidated │ └─► Rebuild PlaneGeometry (41×41 = 1,681 vertices) │ ├─► getTerrainHeight() × 1,681 per chunk │ │ └─► useGameStore.getState().terrainOffsets × 4 lookups │ ├─► getBaseTerrainHeight() × 1,681 (2× noise2D) │ ├─► new THREE.Color() × ~300 (lerp targets) │ └─► computeVertexNormals() per chunk │ │ TOTAL: 49 × (1,681 × 6 + 1,681 × 4 + computeNormals) │ = 49 × ~16,810 operations = ~823,690 operations │ ├─► WaterEngine.updateTerrain() (already optimized: skip if stamp unchanged) │ └─► Minimap BMP regeneration (timer-gated, minor)
🔴 Critical Finding: A single modifyTerrain() call triggers ~824,000 operations synchronously, including 82,369 getTerrainHeight() calls (each doing 4 dictionary lookups + bilinear interpolation) and 49 computeVertexNormals() calls. On Hermes this can block the JS thread for 100-300ms.

Specific Bottlenecks measured

1. terrainOffsets Shallow Copy

// After 50 pats with radius=3:
// ~7×7 = 49 keys per pat × 50 = 2,450 keys
// Object.assign with 2,450 keys: ~0.5ms web, ~3ms Hermes

2. 49 Chunk Geometry Rebuilds

// Chunk.tsx line 62

// ALL 49 chunks rebuild on ANY offset change
// Even chunks 200m away from the modification!

3. getTerrainHeight Store Access

// Called 82,369× per modification event
// Each call: getState() + 4× string template + 4× dict lookup

4. computeVertexNormals × 49

// Chunk.tsx line 60

// Generic Three.js: O(faces × 3)
// 40×40 grid = 3,200 faces × 49 chunks 
// = 156,800 face normal computations

Proposed Fixes planned

Fix 1: Localized Chunk Invalidation

Only rebuild chunks that are geometrically affected by the terrain modification. Check if the modification circle (center ± radius) overlaps the chunk's AABB before invalidating.

Expected savings: From 49 rebuilds → 1-4 rebuilds per modification. ~95% reduction.
Fix 2: Extract terrainOffsets from Zustand → Module-scope Map

Move terrainOffsets out of the reactive Zustand store into a plain Map<string, number> at module scope. This eliminates: (a) the shallow-copy on every modification, (b) the getState() call on every getTerrainHeight() invocation, (c) the Zustand notification cascade.

Expected savings: O(1) mutation instead of O(N) copy. Eliminates 82,369 getState() calls per modification.
Fix 3: Chunk-local terrain dirty flag

Instead of useMemo dependency on terrainOffsets, use a numeric counter per-chunk that only increments when the chunk is determined to overlap the modification area.

📚 Anti-Pattern Catalog

AP-1: GC Storm

Allocating new THREE.Vector3() or new THREE.Object3D() inside useFrame. Each allocation creates GC pressure; on Hermes, GC pauses are 5-50ms.

Fix: Hoist to module scope, reuse via .set().

✅ Fixed in: Beaver, Particles, FloatingLogs, DraggableLogs

AP-2: Zustand Shallow Copy

Using { ...state.largeObject } inside Zustand set() for large state slices. Each spread creates a full shallow copy, triggering O(N) work proportional to object size.

Fix: Move large data out of Zustand into module-scope structures.

⚠️ Partially fixed: batchModifyTerrain batches but still does O(N) copy

AP-3: Broadcast Invalidation

Using a single shared dependency (e.g. terrainOffsets) in useMemo across many components. A single change invalidates ALL components, even those unaffected.

Fix: Compute overlap and only notify affected chunks.

⚠️ TODO: Terrain modification cascade

AP-4: Generic Generic

Using Three.js's generic computeVertexNormals() when the geometry has known structure (e.g. regular grid). The generic version does extra work for arbitrary topology.

Fix: Compute analytical normals from height differences.

✅ Fixed in: WaterRenderer
